### 確定している事実
1. **Crash 順序**: `_B_PRINT`(supervisor) → `_B_SUPER` → exit → `.short 0xFF4C`
   → 0xFFA870 (IPL-ROM `memmove`, `move.l -(a0),-(a1)`) → Address Error
2. **正常順序**: `_B_SUPER` → `_B_PRINT`(user) → exit → 正常終了
3. クラッシュ時 `A1 = BSS start + 1` (奇数) — off-by-one
4. `_B_PRINT` (`trap #15`, IOCS 0x21) 自体は両順序で正常（メッセージは表示される）
5. `0xFFA870` は IPL-ROM 内ルーチン ($FE0000-$FFFFFF)
6. cooperative / preemptive 両方で同一クラッシュ（スケジューラ非依存の可能性）
7. `OBJDIR=obj-$(SCHED)` 分離済み（cooperative/preemptive のビルド成果物混入防止）

### 根本原因の仮説（未確定）
supervisor mode で `trap #15` を呼ぶと SSP が C ランタイムスタックを指し、
user mode だと SSP=old_ssp を指す。この違いが Human68K の PSP/MCB 等のプロセス
管理情報へ与える影響を変え、DOS 終了処理で off-by-one が顕在化する、という仮説。
真因は未特定（PSP/MCB 破壊 / IOCS 内部状態 / BSS アライメント依存の書き込み破壊
のいずれか）。B_SUPER 先順は確実な回避策だが「たまたまアライメントが変わって
潜伏するだけ」の可能性も残る。

## 根本原因解析手順（DIAG_CASE）

`standalone/main.c` の終了シーケンスに `#ifdef DIAG_CASE_N` (N=1..5) で切り替え可能な
診断パスを実装済み。各ケースは **正常順 (B_SUPER 先)** を前提に、B_SUPER 後（user mode）
に何をするかだけを変えて、何がクラッシュ要因かを孤立させる。

### 各ケースが孤立させる変数
| ケース | B_SUPER 後の処理 | 何をテストするか |
|--------|------------------|------------------|
| 1 | なし → `_dos_exit2(0)` | ベースライン。IOCS 呼び出しゼロで終了できるか |
| 2 | `nop×1000` → `_dos_exit2(0)` | 純粋な遅延（IOCS 無し）で悪化するか（タイミング/スタック沈殿） |
| 3 | `_iocs_b_keysns()` → `_dos_exit2(0)` | 文字列引数なしの IOCS `trap #15` が要因か |
| 4 | `_iocs_b_print("A")` → `_dos_exit2(0)` | 短い文字列の B_PRINT が要因か |
| 5 | `_iocs_b_print("A"×16)` → `_dos_exit2(0)` | 文字列長が要因か（BSS/アライメント仮説） |

> 補足: 全ケース B_SUPER 先順なので「正常側」の変動を調べる設計。クラッシュ側
> (supervisor で B_PRINT) は今回の回帰で再確認済み（normal path 修正前 = クラッシュ）。

### ビルド手順
DIAG は variant define を **明示的に併指定** すること（`CDEFINES=` を渡すと
Makefile 内部の `-DSS_BUILD_*` を上書きしてしまうため）。

```bash
. ~/.elf2x68k
cd ssos/standalone
# cooperative で Case N をビルド（N=1..5）
make clean
make SCHED=cooperative CDEFINES="-DSS_BUILD_COOPERATIVE -DDIAG_CASE_N" standalone
# 成果物: ssos_cop.x → xeij で実行、ESC 押下で終了パス(DIAG)が走る
cp ssos_cop.x ~/tmp/
# preemptive で試す場合:
# make SCHED=preemptive CDEFINES="-DSS_BUILD_PREEMPTIVE -DDIAG_CASE_N" standalone
```

> 復帰時は `CDEFINES=` を付けず通常ビルド: `make SCHED=cooperative standalone`

### 実行と観察
xeij で `~/tmp/ssos_cop.x` を起動 → ESC 押下。
観察項目:
- **OK**: メッセージ表示（Case 4/5）後プロンプトに戻る / 何も出ずプロンプトに戻る（Case 1/2/3）
- **CRASH**: "Exceptional Abort By address error" / PC=0xFFA870 / `move.l -(a0),-(a1)`
  / A1 が奇数（BSS+1 系）

### 解釈マトリクス（結果→結論）
- **Case 1 が CRASH** → 要因は B_PRINT ではなく B_SUPER or dos_exit2 or それ以前の状態。
  B_PRINT 要因説を否定。
- **Case 1=OK, Case 2=CRASH** → 純粋な遅延で発現。スタック/タイミング依存。
- **Case 1=OK, Case 3=CRASH** → user mode の IOCS `trap #15` 一般が要因。
  B_PRINT 固有ではなく trap 命令/SSP 切替が鍵（supervisor/user 仮説を強く支持）。
- **Case 1&3=OK, Case 4 or 5=CRASH** → B_PRINT（文字列引数付き IOCS 0x21）固有。
- **Case 4=OK, Case 5=CRASH** → 文字列長依存。BSS/アライメント書き込み破壊説を支持。
- **Case 1..5 全て OK** → B_SUPER 先順は条件下で堅牢。要因は supervisor mode での
  B_PRINT に特定される（残課題は「なぜ supervisor で trap#15 が悪いのか」の解明）。

### 次にやること
1. 上記 5 ケース（cooperative, 必要なら preemptive も）を xeij 実機で実行し、
   OK/CRASH を記録。解釈マトリクスで要因を絞り込む。
2. 要因が supervisor/user の SSP 差に確定したら、`_B_SUPER` 復帰時の SSP/A7 値と
   PSP (`$80` 付近) / MCB の内容をダンプし、off-by-one の物理的発生箇所を特定。
3. 真因修正後、DIAG_CASE コード群と本手順を削除。

### 残課題
