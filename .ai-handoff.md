# AI Handoff

## 現状（s33: preemptive standalone 終了時 Address Error — 調査継続中）

### 修正内容
`_iocs_b_print()` を `_B_SUPER(old_ssp)` の後に移動 (e8a268e)。
応急処置として有効だが、なぜ効くかの説明は不完全。

### 確定している事実
1. Crash: `_B_PRINT` → `_B_SUPER` → `_dos_exit2` → `.short 0xFF4C` → 0xFFA870 (IPL-ROM memmove) → Address Error
2. No crash: `_B_SUPER` → `_B_PRINT` → `_dos_exit2` → 正常終了
3. A1 = `0x0015E429` (奇数) = BSS start + 1 (off-by-one)
4. `_B_PRINT` (`trap #15`, IOCS 0x21) は両方の順序で正常動作 (メッセージ表示される)
5. `_dos_exit2` = `movew sp@(6),sp@-; .short 0xFF4C` — cleanup なし
6. 0xFFA870 は IPL-ROM 内ルーチン ($FE0000-$FFFFFF)
7. "Preemptive"(10文字) と末尾スペース付き(11文字相当) の1バイト差で発現有無が変わる
8. `OBJDIR=obj-$(SCHED)` 分離を併用

### 未確定の仮説
Supervisor mode で trap #15 を呼ぶと SSP=C ランタイムスタック、User mode だと SSP=old_ssq になる。この違いが Human68K の PSP/MCB 等のプロセス管理情報への影響を変え、終了処理で off-by-one が顕在化する。

### 次セッションの診断 (5ケース)

各ケースを standalone/main.c の cleanup 部に実装し、`#ifdef DIAG_CASE_1` 等で切り替えて
xeij でテスト。結果次第で原因を特定できる。
