# SSOS3 アーキテクチャ概説

SSOS3 は X68000 実機およびエミュレータ上で動作する、ベアメタル OS プロトタイプです。

## グラフィックモード設定

SSOS3 では、標準的な IOCS `_CRTMOD` 呼び出しにより画面モードを設定しています。

### IOCS `_CRTMOD` モード一覧

| 分類 | モード番号 | 解像度 | 仮想画面 | 色数 | 備考 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **パレットモード** | 8 | 512 × 512 | 512 × 512 | 256色 | HiReso (31kHz) |
| | **9** | **512 × 512** | **512 × 512** | **256色** | **LoReso (15kHz) / 現在採用中** |
| | 10 | 256 × 256 | 512 × 512 | 256色 | HiReso (31kHz) |
| | 11 | 256 × 256 | 512 × 512 | 256色 | LoReso (15kHz) |
| **High Color** | 12 | 512 × 512 | 512 × 512 | 65536色 | HiReso (31kHz) |
| | 13 | 512 × 512 | 512 × 512 | 65536色 | LoReso (15kHz) |

※ モード 0〜7 は 16色（4プレーン）モード、モード 16〜19 は特殊解像度用です。

## VRAM 構造 (256色モード 9)

X68000 の 256色モードは「パレット形式」ですが、メモリ上では **1画素 = 1ワード (16ビット)** としてマップされます。

- **GVRAM アドレス**: `$C00000`
- **1画素**: 16ビット（下位 8ビットにパレットインデックスを格納）。
- **物理アライメント**: 1ドットが 2バイト を占有するため、`uint16_t*` ポインタによるアクセスが最適です。
- **ライン幅 (Stride)**: 512 ワード (1024 バイト)。
- **座標計算**: `GVRAM + (y * 512) + x`

## パレットと UI カラー (MacOS 7 スタイル)

`_iocs_gpalet` を使用して、MacOS 7 の外観を再現するためのパレット構成を行っています。

- **パレット形式**: `GGGGG RRRRR BBBBB P` (16ビット)
- **カラー構成**:
    - `C_BLACK` (Index 0): 黒。タイトルバーの文字、枠線、横線に使用。
    - `C_WHITE` (Index 1): 白。ウィンドウのコンテンツ背景に使用。
    - `C_GRAY_L` (Index 2): ライトグレー。アクティブウィンドウのタイトルバー、デスクトップ背景（白との交互）に使用。

## デスクトップ UI デザイン

MacOS 7 の視覚的特徴を以下のアルゴリズムで再現しています。

1.  **チェッカーボード背景 (`fill_stipple`)**:
    白とライトグレーを 1ドットおきに交互に描画し、当時の Macintosh 特有のデスクトップ質感を再現。
2.  **タイトルバーの装飾**:
    - **Foreground**: ライトグレー背景に 5本の黒い水平線を描画。中央のタイトル文字部分のみ線をクリッピング。
    - **Background**: 白背景。境界を明確にするため最下段に 1ピクセルの黒線を引く。
3.  **ウィンドウ枠**: すべて 1ピクセルの黒線 (`C_BLACK`) で統一。

## フォント描画

内蔵の **Spleen 5x8** フォント（ASCII 0x20-0x7E）を採用しています。
- **配置**: バイナリ内に 1文字 8バイトのビットマスクとして埋め込み。
- **レンダリング**: 各ビットを走査し、パレットインデックスを GVRAM へ直接書き込み。
- **レイアウト**: 5ピクセル幅の文字に 1ピクセルの隙間を加え、6ピクセル送りで描画。

## 描画パイプライン

### レンダリングモード

メインループの描画フェーズは3つのモードに分かれ、フレームごとに1つだけ実行される。

| モード | 条件 | 処理 |
| :--- | :--- | :--- |
| **Full Redraw** | `need_full == 1` | 全画面 stipple → 全ウィンドウ描画（z-order） |
| **Drag Outline** | `drag >= 0` | 前回 outline 復元 → 新位置保存 → marching ants 描画 |
| **Dirty Content** | それ以外 | 変更のあったテキスト行のみピクセルクリッピング付きで描画 |

### Full Redraw（初回・ドラッグ開始時）

初回フレーム (`need_full = 1`) で画面全体を構築する:

1. `fill_stipple` でデスクトップ全体に白/グレーのチェッカーボードパターンを描画
2. `draw_frame` で各ウィンドウの枠・タイトルバー・コンテンツ背景を z-order 順（下→上）に描画
3. `draw_content_dirty` で各ウィンドウのテキスト内容を描画

このモードは初回フレームと、ドラッグ開始時の対象ウィンドウ矩形のみに限定して呼ばれる。

### Drag Outline（ドラッグ中）

ドラッグ中はウィンドウの実体を描き直さず、marching ants アウトラインのみを更新する:

1. `ol_restore` — 前回保存した outline 位置の元画像を復元（位置変化時のみ）
2. `ol_save` — 新位置の周辺ピクセルをバッファに保存（最大 1200 ワード、位置変化時のみ）
3. `draw_march_outline` — 白黒交互の破線を周長に沿って描画（`frame & 7` でアニメーション）

#### 位置変化検出による最適化

ドラッグ中のフレームは2種類に分かれる:

| フレーム種別 | 処理 | VRAM操作数 |
| :--- | :--- | :--- |
| **位置変化あり** (マウス移動) | restore → save → draw | 1716 ops |
| **位置不変** (マウス停止・アニメのみ) | draw のみ | 572 ops (**67%削減**) |

位置不変時はアウトラインピクセルが同じ周長上にあるため、新パターンで上書きするだけで正しく更新される。バッファの背景データは次の位置変化時まで有効。

#### Fast Path / Slow Path 分岐

`draw_march_outline` / `ol_save` / `ol_restore` の各関数は、ウィンドウが完全に画面内にある場合（通常ケース）に境界チェックを全削除した fast path を使用する:

- **Fast path** (画面内): ポインタ演算による連続アクセス、ループ内分岐なし
- **Slow path** (画面外あり): ピクセル単位の境界チェック（元のロジック）

上辺・下辺は VRAM 内で連続アドレスのため、配列添字 (`GVRAM[y*STR+x]`) ではなくポインタ (`p[i]`) でアクセスし、乗算をループ外に移動。

ドラッグ終了時に `ol_restore` 後、ウィンドウを実描画。

### Dirty Content（通常時・ピクセル単位クリッピング）

通常時は変更のあったテキスト行のみを更新し、全画面再描画を回避する:

```
for each window in z-order (bottom → top):
    for each text line:
        if memcmp(line, prev) != 0:
            draw_text_clip(x, y, line, fg, bg, zpos)
            memcpy(prev, line)
```

**ピクセル単位クリッピング** (`draw_char8_clip`):
各ピクセル描画前に上位ウィンドウの矩形との衝突判定を行い、被覆ピクセルはスキップする:

```
for each pixel (xx, yy) in character glyph:
    for each window above zmap[zpos]:
        if (xx, yy) is inside that window's bounding box:
            skip this pixel  // 上位ウィンドウの描画内容を保持
```

- **利点**: 下位ウィンドウが上位ウィンドウの領域に描画しないため、チラつきなし・上位ウィンドウの再描画不要
- **計算量**: ピクセルあたり最大2回の矩形判定（3ウィンドウ構成）。全体で約200-400回/フレーム。68000 @ 10MHz で無視できるオーバーヘッド

## VRAM 転送アーキテクチャ

### 現在の実装（CPU move.l / move.w 書き込み）

`fill_rect` / `fill_stipple` は `MOVE.L` (32ビット) で2ピクセル同時書き込み。奇数アドレス時は `MOVE.W` で1ピクセルフォールバック。DMA は未使用。

| 関数 | 用途 | 転送量（最悪） | 特徴 |
| :--- | :--- | :--- | :--- |
| `fill_rect` | 矩形塗り潰し | 512×512 = 128K long | move.l で2ピクセル同時書き |
| `fill_stipple` | チェッカーボード背景 | 512×512 = 128K long | 事前計算パターン + move.l |
| `draw_char8` | フォント1文字 | 5×8 = 40 word | fg/bg 同時書き |
| `draw_char8_clip` | クリッピング付きフォント | ≤ 40 word | ピクセル単位で被覆判定 |
| `draw_march_outline` | ドラッグ outline | ≈ 2×(w+h) word | 周長のみ |

### VRAM 書き込みのアドレス計算

256色モードでは1画素 = 1ワード（16ビット）。パレットインデックスを下位8ビットに格納。

```
address = GVRAM_BASE + (y * 512 + x) * 2
        = GVRAM_BASE + y * 1024 + x * 2   (byte address)
```

`uint16_t*` ポインタと stride 512（ワード単位）でアクセス:

```c
#define GVRAM ((volatile uint16_t*)0xC00000)
#define GVRAM_STR 512  /* words per line = 1024 bytes */

pixel(x, y) = GVRAM[y * GVRAM_STR + x]  /* uint16_t write */
```

### パフォーマンス見積もり

68000 @ 10MHz での `MOVE.L #imm, (an)+` は約4サイクル = 400ns/long (2ピクセル分)。

| 操作 | 転送量 | 時間 | 備考 |
| :--- | :--- | :--- | :--- |
| 全画面 stipple | 512×512 = 131,072 long | ≈ 52ms | 事前計算パターン + move.l |
| ウィンドウ1枚 fill | 240×48 = 5,760 long | ≈ 2.3ms | draw_frame 内 |
| テキスト1行 (24文字) | 24×5×8 = 960 word | ≈ 0.38ms | dirty line 更新 |
| 全ウィンドウ dirty | ≈ 2,880 word | ≈ 1.15ms | 通常フレーム |

フレームレートは V-sync (≈57Hz、17.5ms/フレーム) に同期。通常時の dirty 更新は約1.15msでフレーム内に十分収まる。全画面再描画は52msと実時間で約3フレームを要するが、初回およびドラッグ開始時のみ発生。

### move.l 2ピクセル同時書き込み

256色モードでは1画素 = 1ワード。`uint32_t` (long) に2ピクセル分のパレットインデックスを pack して `MOVE.L` で書き込む:

- **fill_rect**: `c2 = (c << 16) | c` を事前計算し、long 書き込みでループ回数半減
- **fill_stipple**: チェッカーボードの2色パターン `pat[0] = (c2 << 16) | c1`, `pat[1] = (c1 << 16) | c2` を行の偶奇で切替。per-pixel 分岐を完全消除
- **奇数アドレス**: x 座標が奇数の場合、最初の1ピクセルを `MOVE.W` で書き込み、以降を `MOVE.L` にフォールバック

### 描画パイプラインの即時反映

dirty content 更新は `frame % 11 == 0` ゲートを削除し、毎フレーム実行。worker thread のデータ更新が次の V-sync で即座に画面に反映される。

### DMA によるさらなる最適化（今後の拡張）

X68000 の DMAC チャネル2（汎用）を使用すると、大面積の VRAM 転送をさらに高速化できる:

- **`fill_rect`**: DMAC のメモリフィルモードで転送。CPU move.l のさらに2-3倍高速
- **`fill_stipple`**: move.l 最適化済み。DMA でもパターン2色の交互転送が必要
- **`draw_char8` / `draw_char8_clip`**: 1文字あたり40ワードと小さく、DMA 起動オーバーヘッドが相対的に大きい。CPUダイレクトが適切

## プリエンティブコンテキストスイッチの 68000/68030 両対応

### 問題: `rte` 命令の例外フレーム差異

68000 と 68030 では `rte` (Return from Exception) が期待するスタックフレームが異なる:

| CPU | `rte` が消費するフレーム |
| :--- | :--- |
| 68000 | SR (2バイト) + PC (4バイト) = **6バイト** |
| 68030 | Format/Vector (2バイト) + SR (2バイト) + PC (4バイト) = **8バイト** |

タイマ割り込みで CPU が自動的に push するフレームは各 CPU 仕様に合致するため、そのまま `rte` で戻れる。しかし、タスクの明示的 yield (s4_task_yield) や新規タスク起動では、コードが手動で SR + PC を push する。68030 でこのフレームを `rte` で消費すると、SR (0x2000) を Format/Vector word と誤認し (format=2 = short bus error)、不正な PC に jump して **bus error** が発生する。

### 解決策: resume_type による復帰方法の選択

TCB の `pad` フィールド (offset 31) を `resume_type` として使用し、3種類の復帰パスを切り替える:

| resume_type | 場面 | 復帰方法 |
| :--- | :--- | :--- |
| 0 | タイマ割り込みによるプリエンプション | `movem.l` → **`rte`** (CPU frame安全) |
| 1 | `s4_task_yield` による自発的 yield | `movem.l` → `move.w (sp)+,%sr` → `move.l (sp)+,%a0` → **`jmp (%a0)`** |
| (新規タスク) | context == stack_base | `move.w #0x2000,%sr` → **`jmp entry`** |

**要点**: `rte` はタイマ割り込みされたタスクのみ使用。yield と新規タスクは `rte` を使わず、手動で SR を復帰して `jmp` で遷移する。これにより CPU 種別に依存しない安全なコンテキストスイッチが実現できる。

### スタックフレーム構造

#### タイマ割り込み (resume_type = 0)

```
SP → d0-d7/a0-a6 (60バイト, movem.l)
     CPU例外フレーム (68000: SR+PC, 68030: Fmt+SR+PC)  ← CPUがpush
```

復帰: `movem.l (sp)+,d0-d7/a0-a6` → `rte` (CPUが自身のフレームを正しく消費)

#### Yield (resume_type = 1)

```
SP → d0-d7/a0-a6 (60バイト, movem.l)
     SR = 0x2000 (2バイト, 手動push)
     .yield_resume アドレス (4バイト, pea)
     元のreturn address (4バイト, bsr s4_task_yield がpush)
```

復帰: `movem.l (sp)+,d0-d7/a0-a6` → `move.w (sp)+,%sr` → `move.l (sp)+,%a0` → `jmp (%a0)` → `.yield_resume` で `rts`

#### 新規タスク

スタックは使用せず、TCB の entry フィールドから直接 jump:

```
move.l  20(a1), a0    | a0 = タスクエントリ関数
move.w  #0x2000, %sr  | 割り込み許可
jmp     (%a0)         | タスク開始
```

## 今後の拡張
- 8x16 フォントおよび全角文字のパレットモード対応。
- DMAC チャネル2を用いた `fill_rect` のさらなる高速化。
- マウス SCC 割り込みハンドラによる IOCS 呼び出し削減。
- `draw_char8_clip` の z-order 最適化 (重なりなし時の fast path)。
- ウィンドウリサイズ・スクロール対応。
